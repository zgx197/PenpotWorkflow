# 基于 Facet 的 AI UI 实现方向

## 文档目的

本文档用于说明：在已经有 `Facet` 这套运行时的前提下，我们后续应如何推进“AI 辅助设计 UI 并快速放入 Godot”的实现方向。

这里的重点不是做一套抽象概念，而是确定一条现实可落地的技术路线。

## 要解决的真实问题

我们后续真正想做的，不是“让 AI 帮我们画几张图”，而是：

- 让设计意图可以结构化表达
- 让 AI 能基于结构化输入生成可运行的页面草稿
- 让页面草稿进入 Godot 时不需要全部重写
- 让设计、程序、AI 使用同一套页面和组件语言

换句话说，我们要解决的是“设计到运行时”的缩短，而不是只解决“设计到图片”的缩短。

## 先回答一个关键选择

有人提出过一条思路：

```text
Figma -> 导出数据 -> AI 生成 H5 -> H5 转 Godot
```

这条路不是完全错误，但对我们来说，不应该成为核心路线。

更适合我们的路线是：

```text
Penpot / Figma
    -> 设计结构导出
    -> Facet UI Schema
    -> Facet Runtime
    -> Godot 页面草稿
    -> 人工细化
```

也就是说，`H5` 可以作为预览手段，但不应该作为核心中间协议。

## 为什么不把 H5 当核心中间层

### 1. H5 适合预览，不适合成为 Godot 的主中间语言

网页语义和 Godot UI 语义有相似之处，但并不等价。

常见问题包括：

- CSS 布局和 Godot 容器布局并非一一对应
- 网页对文本排版和盒模型的处理和 Godot 不同
- 鼠标优先的网页交互不等于游戏 UI 的焦点和手柄导航
- 浏览器容易实现的表现效果，Godot 往往要通过主题、节点和脚本配合

所以 `H5 -> Godot` 这一步很容易在 demo 层面显得顺畅，在工程层面变得脆弱。

### 2. Facet 已经有更合适的承接层

Facet 已经提供了：

- 页面定义
- 动态布局 schema
- 稳定节点 key
- 绑定作用域
- Lua 行为桥
- 页面 runtime

这意味着我们没有必要先把设计描述转换成网页语义，再从网页语义回译为 Godot 语义。

### 3. H5 会把问题从“设计协议”转成“网页翻译”

如果把 H5 当核心中间层，团队很容易开始花大量时间解决这些次要问题：

- HTML 节点和 Godot 节点怎样映射
- CSS 哪些规则可以转，哪些不能转
- 前端脚本逻辑是否要转换成 Lua 或 C#
- 浏览器预览和 Godot 实际效果差异如何收敛

这些问题本质上都不是我们的最终目标。

我们的最终目标是：稳定地把设计意图送进 Facet 和 Godot。

## 更合理的核心路线

推荐把整体实现方向定义为：

```text
设计工具
    -> 导出结构化设计数据
    -> 归一化处理
    -> AI 生成或修正 Facet UI Schema
    -> Facet 生成页面树
    -> Binding / Lua 接入行为
    -> Godot 中预览和迭代
```

这条路线下，AI 的角色不是“直接把图片画成网页”，而是：

- 理解结构
- 补全映射
- 生成 schema
- 生成页面定义
- 生成组件定义
- 辅助生成 Binding 和行为脚本草稿

## 我们应该正式引入的中间层

建议把后续系统的核心中间层命名为：

- `Facet UI Schema`

它至少应该覆盖三类对象。

### 1. Page Schema

描述一个页面的元信息，例如：

- `pageId`
- `layer`
- `layoutType`
- `layoutRef`
- `controllerScript`
- `cachePolicy`
- 页面入口参数定义

### 2. Layout Schema

描述页面或组件的结构树，例如：

- 节点 key
- 节点类型
- 容器关系
- 布局约束
- 文本默认值
- 图片或图标引用
- 可见性默认值
- 子节点

### 3. Component Schema

描述页面内可复用结构，例如：

- 组件 id
- 组件布局根
- 暴露的槽位
- 允许绑定的节点 key
- 可配置变体

如果没有正式的 Component Schema，系统很容易退化成“每页一棵独立大树”，后续复用能力会很差。

## 推荐的整体链路

### 阶段 1. 设计工具导出

先不追求全自动接入 Penpot 或 Figma 的全部能力，而是要求设计导出至少能拿到这些数据：

- 页面和 Frame 层级
- 节点名称
- 节点类型
- 文本内容
- Auto Layout 或类似布局信息
- 间距和尺寸
- 图片资源引用
- 组件实例和变体信息

这一步的目标不是还原所有设计能力，而是先拿到足够驱动结构页面的最小数据集。

### 阶段 2. 设计数据归一化

设计工具导出的原始数据不应直接进入 Facet。

中间应有一层归一化，把不同设计工具的输出整理成统一表示，例如：

- 统一节点类型词汇表
- 统一布局约束描述
- 统一文本节点格式
- 统一图片资源描述
- 统一组件实例引用格式

这一层越稳，后续 AI 和生成器越容易工作。

### 阶段 3. AI 生成或修正 Facet Schema

AI 在这里最适合做的不是自由生成，而是约束内转换。

推荐的工作模式是：

- 输入：归一化设计结构
- 输出：Facet Page Schema + Layout Schema + Component Schema
- 规则：只能使用被允许的节点类型和字段

这样做比让 AI 从截图直接自由输出 Godot 场景可靠得多。

### 阶段 4. Facet 生成页面骨架

Facet 根据 schema 生成：

- 页面定义
- 节点树
- 稳定 key
- 组件插槽
- 初始文本和可见性

这一步产出的应该是“可打开、可绑定、可继续演进”的页面草稿，而不是最终 UI。

### 阶段 5. 绑定和行为接入

页面骨架生成后，再进入：

- Projection 映射
- Binding 注册
- Lua 行为脚本
- 页面跳转关系
- 红点和列表逻辑接入

也就是说，结构层和行为层要故意分阶段，不要企图一次自动完成。

### 阶段 6. 人工细化和验证

最终仍然需要程序和设计共同细化：

- 主题样式
- 动画和过渡
- 输入和焦点
- 手柄导航
- 性能和资源策略
- 异常状态与边界情况

自动化要解决的是 60% 到 80% 的重复结构工作，而不是幻想 100% 无人接管。

## Facet 后续应优先补强的能力

为了让这条路线可落地，Facet 本身要继续往前补几层能力。

### 1. 外部文件化的页面和布局定义

当前很多定义还是内置构造。后续应该支持从外部资源读取，例如：

- `pages/*.json`
- `layouts/*.json`
- `components/*.json`

先不纠结格式一定是 JSON 还是别的，关键是不能一直停留在内存写死。

### 2. 扩展 Layout Schema

当前动态节点类型和样式字段过少。下一步建议优先补：

- `TextureRect`
- `NinePatchRect`
- `GridContainer`
- `TabContainer`
- `RichTextLabel`
- `LineEdit`
- 自定义组件节点占位

布局字段也要逐步增加：

- 对齐方式
- 锚点与偏移
- 主题样式引用
- 图片资源引用
- 颜色与字体 token
- 状态变体引用

### 3. 正式的组件层协议

组件不应该只是“某棵子树可以复用”，而应该成为正式资源层。

至少要能表达：

- 组件根
- 插槽
- 可绑定节点
- 默认变体
- 可替换子内容

### 4. 设计 token 层

如果未来要让 AI 和设计工具稳定协作，就不该让每个页面都直接写死颜色和间距。

建议补一层 token：

- 颜色 token
- 间距 token
- 圆角 token
- 字体 token
- 阴影 token
- 状态样式 token

这样 AI 输出的是“结构 + token 引用”，而不是一堆分散的硬编码视觉数值。

### 5. 导入校验器

Facet 需要有一层离线或编辑期校验，至少验证：

- pageId 是否重复
- 节点 key 是否重复
- 节点类型是否合法
- 绑定目标是否存在
- 模板槽位是否正确
- 控制器脚本是否存在

没有校验器，AI 生成链路会很快被无效数据拖垮。

## 推荐的产品边界

为了避免范围失控，建议明确这条路线当前不追求的内容。

### 当前不追求

- 自动生成完整游戏 UI 运行时逻辑
- 自动处理所有 Godot 主题细节
- 自动完成所有动画和过渡
- 自动推断复杂手柄导航策略
- 自动从截图直接无约束生成正式页面

### 当前重点追求

- 结构化页面快速搭建
- 页面和组件 key 稳定生成
- 设计结构到 Godot 结构的稳定映射
- Binding 和 Lua 接线的半自动化
- 导入结果可预览、可调试、可修正

## 推荐的第一阶段实现范围

如果只做一个现实可落地的 `v0.1`，我建议把范围控制在下面这几件事。

### v0.1 目标

- 支持从设计结构生成简单页面和简单组件
- 支持文本、按钮、容器、图片和列表骨架
- 支持生成稳定节点 key
- 支持输出 Facet 页面定义和布局定义
- 支持在 Godot 中通过 Facet 打开并预览

### v0.1 页面类型

优先选择这些页面做试点：

- 设置页
- 信息面板
- 商店列表页
- 图鉴页
- 任务页

这些页面结构化强、动画要求相对可控，更适合先验证链路。

### v0.1 不建议先碰

- 战斗 HUD
- 复杂拖拽背包
- 强动画页面
- 高度依赖手柄焦点导航的复杂菜单

这些页面对运行时细节要求更高，不适合拿来做第一批导入试点。

## 推荐的推进阶段

### 阶段 A. 固定协议

先定义清楚：

- Page Schema
- Layout Schema
- Component Schema
- Token Schema
- 设计节点到 Facet 节点的映射规则

没有这一步，后面做的导入器和 AI 提示词都会反复返工。

### 阶段 B. 做最小导入器

做一个最小导入器，把设计工具导出的规范化数据转成 Facet schema。

这个导入器不必一次覆盖全部设计能力，但必须保证输出稳定、可校验。

### 阶段 C. 做最小 AI 转换器

让 AI 在受限约束下补：

- 节点类型映射
- 组件抽取建议
- 文本和资源引用整理
- 布局 schema 修正

注意这里的 AI 更适合做“受限转换和补全”，而不是全自由设计。

### 阶段 D. 接 Facet Runtime 预览

把生成结果直接挂到 Facet 页面对外打开，让设计和程序都能在 Godot 中看结果，而不是停在文档层。

### 阶段 E. 补编辑期校验和对照工具

最后补：

- 导入错误列表
- 节点 key 浏览
- 页面结构树查看
- 设计数据和生成结果的差异对照

这一步能极大提升日常使用价值。

## 对当前仓库的直接启发

对 `design-to-facet` 这个仓库来说，后续不应只写“如何从设计工具导出 PNG 给程序”，而应逐步增加这类内容：

- 设计节点命名规则
- 页面 key 规则
- 组件 key 规则
- 设计结构到 Facet 节点类型的映射表
- 哪些设计元素可以自动导入
- 哪些设计元素必须人工处理
- 导入前检查清单
- 导入后验证清单

也就是说，仓库的未来不只是“工作流文档库”，而是“Facet UI 协议和设计交付规范库”。

## 当前结论

在已有 Facet 的前提下，我们最合理的实现方向不是：

- 把 Penpot 或 Figma 设计稿先转成 H5，再想办法翻译成 Godot

而是：

- 把设计结构稳定地转换成 `Facet UI Schema`
- 让 Facet 直接承接结构生成
- 再通过 Binding、Lua 和人工细化完成页面落地

可以把这条路线概括成一句话：

不是“让 AI 生成网页，再把网页转成 Godot”，而是“让 AI 生成 Facet 能理解的页面协议，再让 Godot 直接消费它”。
