# 项目概览

## 仓库定位

`design-to-facet` 是一个面向 Facet 与 Godot 的 UI 设计工作流仓库，目标是为游戏项目提供一套清晰、开源、可复用的设计到运行时协作方式。

它主要回答四个问题：

- 用什么工具表达页面结构和视觉意图
- 设计结果怎样整理成 Facet 可以承接的页面协议
- Facet 怎样把这些协议转成 Godot 可运行的页面骨架
- 团队怎样用统一约定减少返工

## 为什么不再围绕单一设计工具命名

这个仓库不应再只围绕 Penpot 命名，因为后续很可能同时接入：

- Penpot
- Figma
- AI 生成的结构化 UI 描述

它们都只是输入侧工具，不是这套方法的核心承接层。真正决定这条链路能否落地的，是设计结构如何进入 Facet，再进入 Godot。

## 为什么以 Facet 为承接核心

Facet 的价值主要在于：

- 已经具备正式页面 runtime
- 有页面定义、布局来源、节点 key、Binding 和 Lua 行为桥
- 可以同时承接手工场景、模板布局和动态生成布局
- 能为后续 AI 辅助导入提供稳定运行时边界

对于游戏项目来说，Facet 不是设计工具，而是设计结果进入 Godot 之前最关键的承接层。

## 适合解决的问题

- 线框图阶段快速比对多个页面结构方案
- 把复杂 HUD、面板、弹窗拆成可复用组件
- 将设计稿中的状态变化整理成统一页面和组件语言
- 给 Godot 实现阶段提供更明确的节点结构、尺寸、层级和资源说明
- 为后续 AI 生成页面骨架和导入 schema 做准备

## 不适合解决的问题

- 直接生成完整可运行的 Godot UI 逻辑
- 替代 Godot 中的动画、输入、响应式布局和实际运行时细节
- 用一个外部设计工具完整表达所有页面行为和交互边界

## 推荐工作边界

设计工具负责：

- 页面草图
- 状态稿
- 组件结构说明
- 视觉规格
- 资源导出说明

Facet 负责：

- 页面定义
- 布局 schema
- 节点 key 与组件约定
- Binding 承接
- Lua 行为宿主
- 运行时诊断与热重载

Godot 负责：

- `Control` 节点树最终实现
- 运行时交互逻辑
- 输入适配
- 动画与过渡
- 多分辨率适配
- 性能与可用性调优

## 推荐演进路径

1. 先写清文档、命名规范和页面 / 组件约定
2. 再沉淀页面模板、组件模板和节点映射规则
3. 然后为常见设计导出动作补归一化与校验脚本
4. 最后再考虑半自动导入到 Facet 的辅助工具

这个顺序的原因很直接：在协议尚不稳定时，自动化只会固化混乱。先统一页面语言和交付物，再考虑提效。
