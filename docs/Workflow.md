# 设计工具到 Facet / Godot 的工作流

## 总体原则

先定义设计意图，再进入运行时实现。设计工具负责“页面应该怎样组织和表达”，Facet 负责“页面结构怎样被正式承接”，Godot 负责“页面最终怎样运行”。

## 阶段划分

### 1. 需求拆分

在开始画界面前，先明确：

- 这个界面的核心目标是什么
- 玩家在这个界面里要完成什么操作
- 有哪些关键状态和异常状态
- 哪些信息必须优先展示

如果这一步不清楚，后面的原型再精美也没有意义。

### 2. 线框原型

使用 Penpot、Figma 或其他设计工具输出低保真方案，重点验证：

- 信息架构
- 区块层级
- 主次操作顺序
- 弹窗与页面跳转关系

这个阶段不要过早纠结视觉细节。

### 3. 组件抽取

当多个页面出现重复结构时，开始抽取组件，例如：

- 标题栏
- 标签切换
- 资源条
- 背包格子
- 列表项
- 弹窗按钮组

组件一旦抽取，就要补最基本的命名、状态说明和稳定节点 key 约定。

### 4. 高保真与状态稿

在线框稳定后，再补视觉与状态变化：

- 默认态
- Hover / Focus / Pressed
- 禁用态
- 数值异常态
- 空状态
- 加载态

对于游戏 UI，状态稿比单张静态图更重要，因为程序实现时最容易遗漏边界情况。

### 5. 结构化交付给 Facet

交付物建议至少包括：

- 页面总览图
- 关键组件清单
- 状态稿
- 尺寸与间距说明
- 资源导出列表
- 交互说明
- 页面与组件的结构化命名约定

如果后续要进入 AI 或半自动导入链路，还应补：

- 页面 id
- 组件 id
- 节点 key 约定
- 页面层级与容器关系
- 哪些元素需要映射为 Facet 可承接节点

如果只给一堆 PNG，程序和工具基本上还是得重新猜一遍设计意图。

### 6. Facet 承接页面协议

Facet 侧建议按下面的方式理解设计结果：

- 页面 = 一份 `pageId` 驱动的页面定义
- 组件 = 可复用的布局片段与绑定边界
- 节点 = 具备稳定 key 的结构单元
- 状态稿 = Binding、主题、动画或控制器状态的一部分
- 资源导出 = 纹理或装饰资源，不承载页面结构逻辑

不要把“设计图上的结构”机械翻译成“运行时节点的最终结构”，而是要结合 Facet 的页面 runtime、Binding 和组件边界做二次整理。

### 7. Godot 实装

Godot 侧建议按下面的方式理解 Facet 页面：

- 页面 = 一个或一组 `Control` 场景与运行时页面壳子
- 组件 = 可复用 UI 子场景或动态布局片段
- 行为 = Lua 控制器或 C# 生命周期逻辑
- 视觉实现 = 主题、动画、纹理、输入与适配

Facet 负责承接协议，Godot 负责最终运行时质量。

## 页面命名建议

建议格式：

```text
[模块]-[页面]-[状态]
```

示例：

- `meta-main-default`
- `inventory-panel-empty`
- `dungeon-hud-combat`
- `shop-dialog-confirm`

## 组件命名建议

建议格式：

```text
[域]-[组件]-[变体]
```

示例：

- `hud-resource-bar-primary`
- `inventory-slot-rare`
- `button-action-danger`

## 节点 key 建议

建议格式：

```text
[页面或组件]-[区域]-[元素]
```

示例：

- `inventory-header-title`
- `shop-item-list`
- `dialog-action-confirm-button`

节点 key 应优先服务于 Facet 的稳定绑定和结构承接，而不是只服务于设计文件中的临时命名。

## 资源交付建议

### 应导出的内容

- 图标
- 装饰性位图
- 特殊形状切片
- 程序不方便实时绘制的复杂视觉元素

### 不建议导出的内容

- 纯文字
- 可由 Godot 直接完成的基础色块
- 布局本身
- 本应交由主题系统控制的常规按钮状态

如果把太多基础元素都导成图片，后面适配分辨率和主题时会非常痛苦。

## 评审检查清单

在进入 Facet / Godot 实现前，至少检查以下内容：

- 页面名称是否统一
- 组件是否已抽取而不是重复绘制
- 状态稿是否覆盖主要交互路径
- 是否标记出哪些元素必须导出为图片
- 是否说明了尺寸、对齐和间距原则
- 是否明确了页面切换与弹窗唤起关系
- 是否补齐页面 id、组件 id 和关键节点 key

## 最小可执行工作流

如果项目还很早期，先做到这五步就够了：

1. 用设计工具做页面线框
2. 为关键页面补状态稿
3. 输出一份组件、命名和导出清单
4. 约定页面 id、组件 id 和关键节点 key
5. 在 Facet / Godot 中按清单实现，而不是凭单图猜测

这套流程不复杂，但足以显著减少 UI 返工，并为后续 AI 辅助导入做好准备。
